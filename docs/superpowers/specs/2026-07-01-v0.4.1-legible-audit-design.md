# v0.4.1 — "Legible audit" — design

**Milestone:** `v0.4.1 — Legible audit`
**Date:** 2026-07-01
**Status:** approved design → implementation
**Branch base:** `origin/main` (must include the v0.4.0 release, commit `81771b2`)

## Thesis

v0.4.0 gave the audit skill a rigorous *method* (two-pass presence/omission,
omission table, adopted-principle calibration) and a reproducible report *header*
(`report-format: v1`, from #28). Its *findings output* did not keep up: bare
`P#` codes, an optional/variable table, and a non-deterministic artifact
(`plumb-line-audit.md` written sometimes, not others).

v0.4.1 brings the output to the same rigor as the method — **a legible,
deterministic report** — plus two riders that were scoped into the milestone: a
runtime totality parity bug (#80) and a README on-ramp fix (#86).

The audit skill is epistemic-honesty tooling; an inconsistent report shape and an
inconsistent artifact are small self-violations of the very properties (P8
reproducibility, P6 maturity honesty) it demands of the code it reviews. Fixing
them is dogfooding.

## SemVer decision (recorded, not implicit)

- Release version: **`0.4.0 → 0.4.1` (patch).** #83/#84 carry the `enhancement`
  label, but they change only the **skill surface** (the report a reviewer reads)
  — not the published `plumb-line-provenance` npm/PyPI API. The repo's rule
  ("feature → minor") is about the *published packages*; a skill-output
  refinement plus a bugfix is a patch. #80 (the only published-code change) is a
  bugfix. Net: **patch.**
- `PROVENANCE_VERSION` (envelope wire): **unchanged.** No metadata-format change.
- `report-format` (skill report convention, independent third number):
  **`v1 → v2`.** The report shape changes materially (glossary + mandatory
  table); a v1 and a v2 report no longer share a structure, so the honest move is
  a bump, not a silent v1 mutation.

## Component 1 — Audit report contract (skill)

**File:** `skills/plumb-line-audit/SKILL.md`, the `## Report (audit format)`
section (currently main lines 83–99). The #28 header block is retained; only its
version string changes (`v1 → v2`) and the findings/glossary contract is added.

### 1a. Principle glossary + inline names (#83)

- After the header block, before any findings, the report emits a **glossary
  block**: one line per principle referenced, `P# — <name>` + a short gist.
  Canonical names come from `reference/portable-principles.md` (source of truth —
  do not invent names; read them from the file).
- Every `P#` reference anywhere in the report renders with its name inline
  (`P3 — laundered uncertainty`), never a bare `P3`.
- Skill rule added: **explain before use** — a principle is named at or before
  its first reference.
- **Acceptance (#83):** a run names each referenced principle at least once
  before/at first reference; no bare `P#` without an adjacent name.

### 1b. Canonical findings table (#84)

- The findings section is **always** the structured table — no freeform-prose
  fallback, no run-to-run shape variance.
- **Columns (fixed order):**

  `Path` | `Line` | `Function` | `Issue` | `Suggested Fix` | `Principle`

  - `Path` — repo-relative (replaces ambiguous "Site").
  - `Line` — line number (or range).
  - `Function` — enclosing function/symbol if applicable, else `—`.
  - `Issue` — one-line description of the finding.
  - `Suggested Fix` — at-a-glance remedy (a direction, not a patch),
    immediately after `Issue` per the ticket.
  - `Principle` — the inline-named principle (`P3 — laundered uncertainty`),
    satisfying #83 in-row.
- **Clean result:** still the canonical shape — header + glossary (may be empty
  if nothing referenced) + an explicit "no findings" statement. A clean result
  is a valid result.
- **Omission-pass table** (the existing per-output enumeration table in the
  Method section) is a separate artifact from this findings table and is
  unchanged by this work, except that its principle references also gain inline
  names.
- **Acceptance (#84):** repeated runs on the same input produce the same report
  *shape* (always the canonical table); Path/Line/Function render separately; a
  Suggested Fix per finding.

### 1c. Deterministic artifact — "always offer, never auto-write" (#85)

- Every run **prints the full report to chat** (header + glossary + table +
  summary count).
- The skill **never auto-writes** `plumb-line-audit.md`. After printing, it
  **explicitly offers** to write the report to `plumb-line-audit.md` and writes
  only on explicit user confirmation.
- One contract sentence is added to the Report section stating this, so the
  behavior cannot drift.
- **Acceptance (#85):** repeated runs on the same input produce the same
  file-write behavior (never write without an explicit yes).

### 1d. report-format bump (skill)

- Header key `report-format: v1` → `report-format: v2` in the skill spec.
- The bootstrap skill's step-5 report (`skills/plumb-line-bootstrap/SKILL.md`)
  shares the header convention; check it and bump/align in lockstep if it also
  emits a `report-format` string, so the two skills don't diverge on the version.

## Component 2 — `audit_meta` totality parity (#80, published)

**Files:** `primitives/python/audit.py`, `primitives/conformance/cases.json`,
both conformance suites.

- **Bug:** Python `audit_meta` guards `if meta is None: return ['missing meta']`,
  so `audit_meta(0)` / `''` / `False` falls through to `meta.get(...)` and raises
  `AttributeError`. SPEC §5 requires the checker be **total** (never throw). JS
  `auditMeta` returns `["missing meta"]` for any falsy input.
- **Fix:** align the Python guard to JS totality — treat any **non-dict** `meta`
  (not just `None`) as `["missing meta"]`, mirroring the existing
  `validate_envelope` guard shape (`meta is None` / `not isinstance(meta, dict)`).
- **Parity contract (TDD, failing test first):** add an `audit`-kind row to
  `primitives/conformance/cases.json` pinning a falsy-scalar input (e.g. `0`)
  with expected output `["missing meta"]`. This row must **fail the Python suite
  before the fix** and pass both suites after — that failure is the proof.
- **Acceptance (#80):** `audit_meta(0)` returns `["missing meta"]` (does not
  throw); the new conformance case passes in both JS and Python;
  `node primitives/conformance/report.mjs` exits zero.

## Release mechanics

1. Implement on a branch off `origin/main` (this worktree already is).
2. **#80 first, TDD:** add failing conformance case → fix Python → both suites +
   parity gate green.
3. Skill changes (#83/#84/#85/#1d) + README (#86).
4. Update `examples/AUDIT-EXPECTATIONS.md` / harness fixtures for
   `report-format: v2` (per #84 acceptance — fixtures updated if the format
   version changes).
5. **Run `docs/release-harness.md`** before tagging: the diff touches `skills/`
   *and* `primitives/`, so blind validation + dogfood self-audit are required and
   recorded as dated sections. A validation FAIL (missed planted violation)
   blocks the tag; dogfood findings follow fix-or-defer → `audit-deferral` issue.
6. Version bump: `node scripts/bump-version.mjs 0.4.1` (3 manifests + CHANGELOG
   `[Unreleased] → [0.4.1]` in lockstep). Do **not** touch `PROVENANCE_VERSION`.
7. Open the release PR; human merges; then explicit "go" for the tag push
   (tag-triggered publish only — never hand-publish).
8. Close #80, #83, #84, #85, #86 on merge/release.

- `cd primitives/js && npm test` — conformance parity for the new case.

## Risks / notes

- **Harness fixtures vs. format bump:** the blind-validation scoring is about
  *which findings* are caught, not the table's exact columns, so `report-format:
  v2` should not change pass/fail outcomes — but AUDIT-EXPECTATIONS references to
  the report shape must be swept so a validator scoring "shape" isn't left on v1.
- **Bootstrap/audit version drift:** if only the audit skill bumps to v2 and the
  bootstrap report stays v1, the two skills diverge. Component 1d handles this by
  checking bootstrap in the same change.
- **Clean-run legibility:** the glossary could be noise on a clean repo. Mitigate
  by emitting only the principles actually referenced (empty/omitted when clean),
  keeping a clean report short.
